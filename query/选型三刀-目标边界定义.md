# 项目思考：选型三刀，目标边界定义

> 时间：2026-03-20
> 人物：黑爪爪 & 用户

---

## 用户问题

Moonshine AI 二次开发方案已出，Electron + moonshine-small，选型方向确认。

**核心问题：这个项目到底要「做到什么程度才算够」？**

---

## 黑爪爪分析

### 第一刀：质疑根

「Electron + moonshine-small」，不是技术选型，是**立场宣言**。

> 选 Electron，是因为愿意用一点运行时开销换一套代码跨三端。
> 选 small，是因为体积和速度的乘积最优。

这是对的。技术选型没有绝对优劣，只有「此刻，这个团队，这个目标」的最优解。

### 第二刀：方案缺了什么

多了「怎么做」，少了一层：

> **做到什么程度才算够？**

### 第三刀：三个必须现在定义的边界

**边界一：推理延迟目标**

moonshine-small 官方延迟 <100ms，但这只是模型推理。

端到端链路：
```
麦克风采集（0.3s buffer）→ 传输 → moonshine（~80ms）→ 输出
= 端到端 380ms
```

0.3s 的 chunk 如果砍到 0.16s，延迟可降至 ~240ms，但推理线程压力翻倍。

→ **核心问题：延迟目标定在 500ms 可接受，还是必须 <300ms？**

**边界二：精度 vs 隐私**

medium 比 small 精度高 10-15%，但体积从 95MB → 245MB，安装包 120MB → 300MB。

→ **核心问题：用户愿不愿意多下 180MB 换 10-15% 精度提升？**

**边界三：说话人分离的代价**

pyannote.audio 需要额外 ~200MB 模型，且需要多声道物理隔离才能真正分离两个人。单声道做声纹分类，那叫「猜」，不叫「分离」。

→ **核心问题：双路分离是物理隔离的双人场景，还是会议多人场景？**

### 结论

在继续开发之前，需要先确认第一版交付标准：

| 目标 | 时间 |
|------|------|
| 「能跑起来，录一段，导出文字」 | 一个周末 |
| 「产品级 UI，打包发布，能给朋友用」 | 2-4 周 |

## 后续补充：moonshine-ai 源码分析（2026-03-21）

官方 GitHub: https://github.com/moonshine-ai/moonshine

### Python API（moonshine-voice）

```python
from moonshine_voice import Transcriber, TranscriptEventListener

class Listener(TranscriptEventListener):
    def on_line_started(self, event): pass
    def on_line_text_changed(self, event):
        print(event.line.text)  # 实时输出
    def on_line_completed(self, event):
        print(event.line.text)  # 最终输出

t = Transcriber(model_path=None, model_arch="small_streaming")
t.add_listener(Listener())
t.start()
t.add_audio(chunk, sample_rate=16000)
t.stop()
```

### 架构优势

- **事件驱动**：`TranscriptEventListener` 原生支持，`on_line_text_changed` 即为实时字幕回调，无需自行实现去重逻辑
- **MicTranscriber**：内置麦克风管理，sounddevice 自动调用
- **流式保证**：`on_line_started` → `on_line_text_changed` → `on_line_completed`，每个段落精确触发
- **多流支持**：一个 Transcriber 实例可绑定多个 Stream（双路时各用一个实例）

### transcribe.py 已重写

- ✅ 包名改为 `moonshine-voice`
- ✅ 使用 `TranscriptEventListener` 事件驱动
- ✅ `on_line_text_changed` → JSON stdout，Electron 直接消费
- ✅ 双路各自独立 Transcriber 实例，避免状态混淆
- ✅ chunk 160ms，端到端 ~260ms ✅
